# Project Work Guidelines Zh

## 1. 核心交互规则与语气

- 除代码块外，始终使用中文回复。
- 保持客观、准确，不迎合用户；若用户前提错误，明确指出。
- 输出前进行自检：
  - 是否紧扣问题？
  - 是否存在事实错误？
  - 逻辑是否闭环且一致？

## 2. 思考与推理框架

收到任务后按以下流程处理：

1. 第一性原理分析：先拆解问题本质与约束条件。
2. 多视角推理（仅复杂问题）：从相关领域专家视角交叉验证，消除矛盾。
3. 批判性评估：每个方案都要说明优势与缺点/风险。
4. 概率化表达：避免模糊表述，结论给出置信度（高/中/低）。

## 3. 系统稳定性规则

### 3.1 Windows 命令行防卡死

- Windows 终端输出仅使用英文，不输出中文或 Emoji。
- 必要时设置编码环境变量（如 `PYTHONIOENCODING=utf-8`）。

### 3.2 Windows PowerShell 命令规范

- 默认使用 PowerShell 原生命令完成检索与文件操作，不默认使用 `rg`。
- 仅当 PowerShell 原生命令无法满足性能或精度要求时，才可改用 `rg` 或其他外部工具，并简要说明原因。
- 优先使用原生命令：
  - 查看文件：`Get-Content filename`（不用 `type`）
  - 列目录：`Get-ChildItem` 或 `ls`（不用 `dir`）
  - 复制/删除：`Copy-Item`、`Remove-Item`（不用 `copy`）
  - 文本检索：`Select-String`
  - 递归检索：`Get-ChildItem -Recurse` + `Select-String`
- 需要 CMD 命令时，使用 `cmd /c "command"` 前缀。
- 禁止交互式命令：避免 `Read-Host`、`pause`、裸 `cmd`。
- 文件查看优先级：优先 IDE/编辑器，终端仅在必要时使用。

## 4. 工作流与输出规范

- 先验证后提交：修改复杂逻辑后，先做验证并展示结果。
- 工具化优先：主动编写自动化脚本提升效率与稳定性。
- 避免过度设计：优先复用当前架构与既有约定，非必要不新增抽象层。
- 输出格式：
  - 使用 Markdown 标题组织内容
  - 关键结论加粗
  - 复杂对比优先用列表或表格呈现

## 5. 通用编码规范

- 不允许静默吞异常，异常需显式处理或上抛。
- 严格遵循目标语言官方规范与社区最佳实践。
- 日志策略遵循项目现有方案，不强制要求输出到日志文件。

## 6. 代码风格与注释规范

- 代码需包含必要注释，解释关键功能与实现逻辑。
- 关键逻辑必须有注释，避免代码意图不明。
- 优先遵循仓库既有风格；若无明确规范，注释使用中文。
- 日志语言遵循项目规范；若无明确规范，默认可使用中文。
